# 第一阶段：需求拆解与系统设计

## 本步目标
- 基于 `产品设计文档V2.0.md` 与 `开发流程.md`，完成可直接进入第二阶段（项目骨架搭建）的系统设计方案。
- 明确模块边界、权限边界、数据边界（校区隔离 + 创建人隔离）与统计口径。

## 实现方案

### 1. 功能模块清单（4.1）
1. 登录与校区选择
   - 账号密码登录（JWT）
   - 销售登录后必须选择当前校区
   - 当前校区进入会话上下文（token/session）
2. 用户管理（管理员）
   - 用户 CRUD、状态管理、角色绑定、可访问校区绑定
3. 角色管理（管理员）
   - 角色 CRUD、权限点配置（RBAC）
4. 校区管理（管理员）
   - 校区 CRUD、启停状态管理
5. 学生管理（销售）
   - 学生 CRUD、同校区重名校验、欠费状态展示
6. 课程信息录入（销售）
   - 学生维度课程录入，支持一次添加多课程
7. 消课管理（销售）
   - 消课记录创建、课时扣减、负课时高亮、历史记录查询
8. 低课时提醒（销售）
   - `remaining_hours < 3` 自动进入提醒范围
9. 仪表盘（老板）
   - 在读人数、总课时、剩余课时、总收入、未收入课时费用
10. 报表管理（管理员/老板）
   - 多维筛选（校区/销售/课程类型/时间）+ 导出
11. 日志管理（管理员/老板）
   - 关键操作日志查询与删除

### 2. 角色权限矩阵（4.2 + 权限细化）

| --- | --- | --- | --- |
| 登录/鉴权 | 登录、登出、刷新 | 登录、登出、刷新 | 登录、登出、刷新 |
| 校区选择 | 可切换（用于查看） | 必选当前校区 | 可切换（用于查看） |
| 用户管理 | CRUD | 无 | 无 |
| 角色管理 | CRUD + 分配权限 | 无 | 无 |
| 校区管理 | CRUD | 无 | 无 |
| 学生管理 | 可查看（可选） | CRUD（仅当前校区且本人创建） | 只读查看（可选） |
| 课程录入 | 可查看（可选） | CRUD（学生名下课程） | 只读查看（可选） |
| 消课管理 | 只读/筛选 | 创建 + 查询（仅当前校区且本人相关） | 只读查询 |
| 低课时提醒 | 查看全局（按权限） | 查看本人当前校区 | 查看汇总（可选） |

| 报表管理 | 查看/导出 | 无 | 查看/导出 |

> - 销售：仅 `current_campus_id` 且 `created_by = current_user_id`

### 4. 核心业务流程（4.3）
1. 登录与上下文建立
   - 用户登录成功 -> 获取角色与可访问校区 -> 销售选择当前校区 -> 写入上下文
2. 学生录入流程
   - 新建学生 -> 同校区重名校验 -> 保存学生 -> 批量写入课程
3. 消课流程
   - 选择学生课程 -> 提交消课 -> 生成消课记录 -> 扣减剩余课时
   - 若 `< 3` 标记低课时；若 `< 0` 标记负课时高亮
4. 报表/仪表盘流程
   - 基于筛选条件聚合统计（service 层）-> 返回前端展示 -> 支持导出
5. 日志流程
   - 对关键增删改与关键查询写入日志（操作者、模块、动作、目标、校区、时间）

### 6. 数据库表设计草案（4.3）
1. `users`
   - `id`、`username`、`password_hash`、`role_id`、`status`、`created_at`、`updated_at`
2. `roles`
   - `id`、`name`、`code`、`permissions(json)`、`created_at`、`updated_at`
3. `campuses`
   - `id`、`name`、`status`、`created_at`、`updated_at`
4. `user_campuses`
   - `id`、`user_id`、`campus_id`、`created_at`
5. `students`
   - `id`、`name`、`phone`、`total_amount`、`paid_status`、`remark`
   - `campus_id`、`created_by`、`created_at`、`updated_at`
   - 约束：`UNIQUE(campus_id, name)`
6. `student_courses`
   - `id`、`student_id`、`course_name`、`course_price`、`total_hours`、`remaining_hours`
   - `course_type`、`remark`、`created_at`、`updated_at`
7. `course_consumptions`
   - `id`、`student_course_id`、`consumed_hours`、`consumption_time`
   - `operator_id`、`remark`、`created_at`、`updated_at`
8. `operation_logs`
   - `id`、`operator_id`、`module`、`action`、`target_id`、`detail`
   - `campus_id`、`created_at`

### 7. 关键字段说明（4.4）
- `paid_status`：学生是否交费（布尔/枚举），用于欠费标签与未收入统计。
- `total_amount`：学生总金额（录入金额），不强制等于课程价格汇总。
- `course_price`：单课程价格，作为收入与未收入费用计算基准。
- `total_hours`：课程总课时（购课课时）。
- `remaining_hours`：课程剩余课时，可为负数（允许超额消课）。
- `consumption_time`：本次消课业务时间，用于核查重复消课与报表筛选。
- `created_by`：学生记录创建销售，作为销售数据隔离关键字段。
- `campus_id`：校区主维度字段，所有核心业务查询必须携带。

### 8. 风险说明与处理策略（4.5）
1. 同校区学生重名
   - 风险：重复建档导致课时分散
   - 方案：数据库唯一约束 + 新建前接口校验双保险
2. 超额消课（负课时）
   - 风险：课时穿透造成误解
   - 方案：允许写入，前端高亮 + 报表单独可筛选
3. 欠费上课
   - 风险：财务口径不一致
   - 方案：业务允许；欠费显著标识；未收入费用独立统计
4. 重复消课
   - 风险：误操作导致重复扣减
   - 方案：不阻断；提供“同课程同时间附近记录提醒”与便捷核对列表
5. 销售跨校区数据泄漏
   - 风险：权限越界
   - 方案：后端强制附加 `campus_id + created_by` 条件，前端仅做辅助限制
6. 统计口径漂移
   - 风险：仪表盘与报表数字不一致
   - 方案：统计逻辑统一放 service 层复用，禁止前端自行拼装
