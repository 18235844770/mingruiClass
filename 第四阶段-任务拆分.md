## 4.1 认证基础（JWT + 用户信息）

| 序号 | 任务 | 说明 / 验收 |

| 4.1.1 | 登录接口 | `POST /api/auth/login`：校验用户名密码，签发 JWT；载荷含 `sub`、`roleCode`（或等价字段） |

## 4.3 当前校区上下文（服务端强制）

> 目标：**不能只靠前端**传校区 ID；后端对「销售」相关**写读**必须校验校区归属。

| 序号 | 任务 | 说明 / 验收 |
| --- | --- | --- |
| 4.3.1 | 约定传输方式 | 二选一或组合：**请求头** `X-Campus-Id`（与现有前端一致）或 **短生命周期** 的「选校区」接口换发含 `campusId` 的 JWT；**推荐先统一 Header**，便于与现有前端对齐 |
| 4.3.2 | 解析与注入 | `CampusContext`（或 `Request` 扩展）：从 Header 解析 `campusId`，校验 UUID 格式 |
| 4.3.3 | 销售校验 | `roleCode === sales` 时：**必须**带 `X-Campus-Id`；且该校区必须在 `user_campuses` 中存在 `(user_id, campus_id)` |
| 4.3.4 | 非销售 | `admin` / `owner`：可选 Header 作为「筛选当前查看的校区」；**无 Header 时**行为需在实现时定稿（如：统计接口要求必选校区或默认全部） |
| 4.3.5 | 守卫 | `CampusGuard`（或组合在业务守卫中）：在销售访问业务接口前执行 4.3.1～4.3.3 |

**产出**：任意销售业务接口在「未选校区 / 非法校区」时返回 **4xx**。

### 4.3 执行结果（已完成）

- [x] 4.3.1 传输方式：统一采用请求头 `X-Campus-Id`。
- [x] 4.3.2 解析注入：新增全局 `CampusGuard`，解析 Header 并校验 UUID，注入 `req.campusId`。
- [x] 4.3.3 销售校验：`roleCode === sales` 时强制要求 Header，且必须命中 `user_campuses` 绑定。
- [x] 4.3.4 非销售：Header 为可选；若提供则校验格式并注入上下文。
- [x] 4.3.5 守卫：`CampusGuard` 已作为全局 Guard 注册；并通过 `SkipCampusCheck` 放行 `auth/me`、`campuses/mine`。
- [x] 验证接口：新增 `GET /api/students` 占位接口用于验证 guard 行为（销售未带或越权校区将返回 4xx）。

## 4.4 销售数据隔离（查询与写入边界）

| 序号 | 任务 | 说明 / 验收 |
| --- | --- | --- |
| 4.4.1 | 查询条件 | 对 `students` 等销售数据：`student.campus_id = 当前校区` **且** `student.created_by = 当前用户 id` |
| 4.4.2 | 封装 | `buildSalesStudentScope(userId, campusId)` 或 Prisma `where` 工厂，**禁止在 Controller 手写散落条件** |

| 4.4.4 | 消课 | `course_consumptions` 通过 `student_course → student` 校验归属与校区（后续阶段与消课模块一起实现） |

**产出**：在**至少一条**真实业务查询（如学生列表占位接口）上可演示隔离。

### 4.4 执行结果（已完成）

- [x] 4.4.1 查询条件：`GET /api/students` 已落地销售数据范围 `campus_id = 当前校区` 且 `created_by = 当前用户`。
- [x] 4.4.2 封装：新增 `buildSalesStudentScope(userId, campusId)`，避免在 Controller 散落手写条件。
- [x] 4.4.3 写入边界：`POST /api/students` 创建学生时，销售账号强制写入 `campus_id = 当前校区`、`created_by = 当前用户`。

## 4.5 管理员 / 老板权限策略（RBAC）

| 序号 | 任务 | 说明 / 验收 |
| --- | --- | --- |
| 4.5.1 | 角色常量 | 与 `roles.code` 对齐：`admin`、`sales`、`owner` |
| 4.5.2 | 路由级控制 | `@Roles('admin')` 等 + `RolesGuard`（或等价），保护 **用户/角色/校区管理**、**全量配置** |
| 4.5.3 | 老板 | 仅开放 **仪表盘、报表、日志**（只读）；**无**学生/消课写接口 |
| 4.5.4 | 管理员 | 可访问系统配置类接口；**是否**跨校区查看学生数据按产品定稿（默认：管理员可跨校区只读或全量，需在接口层显式分支） |

**产出**：未授权角色访问管理接口返回 **403**。

### 4.5 执行结果（已完成）

- [x] 4.5.1 角色常量：新增 `admin`、`sales`、`owner` 角色代码常量并统一复用。
- [x] 4.5.2 路由级控制：`RolesGuard` 作为全局 Guard 启用；管理接口加 `@Roles('admin')`。
- [x] 4.5.3 老板能力：新增老板只读占位路由（仪表盘/报表/日志）并限制为 `owner` 访问。
- [x] 4.5.4 管理员能力：用户/角色管理占位路由限制为 `admin`；学生接口限制为 `admin`/`sales`（老板不可写学生）。
